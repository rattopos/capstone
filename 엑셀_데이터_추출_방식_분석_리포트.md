# 엑셀 데이터 추출 방식 분석 리포트

## 분석 대상 파일

- `src/excel_extractor.py`: 엑셀 데이터 추출 모듈
- `src/word_template_filler.py`: **삭제됨** (워드 출력 모듈 제거로 인해 더 이상 사용되지 않음)

## 현재 데이터 추출 방식

### 1. 주 방식: **절대 좌표 기반**

현재 시스템은 **절대 좌표(셀 주소) 기반**으로 엑셀 데이터를 추출합니다.

#### 구현 세부사항

**`src/excel_extractor.py`**:
- `parse_cell_address(cell_address: str)`: 셀 주소 문자열(예: 'A1', 'B5')을 행/열 인덱스로 변환
- `get_cell_value(sheet_name: str, cell_address: str)`: 특정 셀의 값을 절대 좌표로 가져옴
- `get_cell_range(sheet_name: str, range_address: str)`: 셀 범위(예: 'A1:A5')의 값을 가져옴

**템플릿 마커 형식**:
- 단일 셀: `{시트명:A1}`
- 셀 범위: `{시트명:A1:A5}`
- 계산식 포함: `{시트명:A1:A5:sum}`, `{시트명:A1:A2:증감률}`

**`src/template_filler.py`의 `process_marker()` 메서드** (라인 1129-1180):
```python
# 셀 주소 형식 확인 (라인 1129)
if not re.match(r'^[A-Z]+\d+', cell_address):
    # 헤더 기반 키인 경우 (동적 마커 처리)
    ...
else:
    # 셀 주소 형식인 경우 - 절대 좌표로 직접 추출
    raw_value = self.excel_extractor.extract_value(marker_sheet_name, cell_address)
```

### 2. 보조 방식: 헤더/키워드 기반 (제한적 지원)

시스템은 헤더 기반 매핑도 일부 지원하지만, 이는 **보조 기능**이며:

- `FlexibleMapper` 클래스를 통해 헤더 기반 컬럼 찾기 지원
- `SemanticMarkerResolver`를 통한 동적 마커 처리 (예: `{시트명:전국_증감률}`)
- 하지만 이들도 결국 **헤더를 찾아서 셀 주소로 변환**한 후 추출

**핵심 문제**: 헤더 기반 키워드도 최종적으로는 `excel_extractor.extract_value()`를 호출하여 절대 좌표로 변환된 후 추출됩니다.

## 절대 좌표 기반의 취약점

### 1. 데이터 행 추가/삭제 시 문제점

#### 시나리오 1: 헤더 행 위에 행 추가
```
원본 구조:
  행 1: 헤더 (A1, B1, C1, ...)
  행 2: 데이터 (A2, B2, C2, ...)
  행 3: 데이터 (A3, B3, C3, ...)

템플릿 마커: {광공업생산:A2}  ← 첫 번째 데이터 행

행 추가 후:
  행 1: 새 제목 행 추가
  행 2: 헤더 (A2, B2, C2, ...)  ← 헤더가 한 행 아래로 이동
  행 3: 데이터 (A3, B3, C3, ...)  ← 원래 A2였던 데이터가 A3로 이동
  행 4: 데이터 (A4, B4, C4, ...)  ← 원래 A3였던 데이터가 A4로 이동

결과: {광공업생산:A2}는 이제 헤더를 가리키게 됨 (잘못된 데이터)
```

#### 시나리오 2: 데이터 행 삭제
```
원본 구조:
  행 1: 헤더
  행 2: 전국 데이터 (A2)
  행 3: 서울 데이터 (A3)
  행 4: 부산 데이터 (A4)

템플릿 마커: {광공업생산:A3}  ← 서울 데이터

행 삭제 후 (전국 행 삭제):
  행 1: 헤더
  행 2: 서울 데이터 (A2)  ← 원래 A3였던 데이터가 A2로 이동
  행 3: 부산 데이터 (A3)  ← 원래 A4였던 데이터가 A3로 이동

결과: {광공업생산:A3}는 이제 부산 데이터를 가리킴 (서울 데이터가 아님)
```

#### 시나리오 3: 중간 행 삽입
```
원본 구조:
  행 1: 헤더
  행 2: 전국 (A2)
  행 3: 서울 (A3)
  행 4: 부산 (A4)

템플릿 마커: {광공업생산:A3}  ← 서울 데이터

중간에 "합계" 행 추가:
  행 1: 헤더
  행 2: 전국 (A2)
  행 3: 합계 (A3)  ← 새로 추가된 행
  행 4: 서울 (A4)  ← 원래 A3였던 서울 데이터
  행 5: 부산 (A5)  ← 원래 A4였던 부산 데이터

결과: {광공업생산:A3}는 합계 데이터를 가리킴 (서울 데이터가 아님)
```

### 2. 열 추가/삭제 시 문제점

```
원본 구조:
  A열: 지역명
  B열: 2023 1/4
  C열: 2023 2/4
  D열: 2023 3/4

템플릿 마커: {광공업생산:D3}  ← 2023 3/4 분기 서울 데이터

중간에 열 추가 (2023 1.5/4 추가):
  A열: 지역명
  B열: 2023 1/4
  C열: 2023 1.5/4  ← 새로 추가된 열
  D열: 2023 2/4  ← 원래 C였던 데이터
  E열: 2023 3/4  ← 원래 D였던 데이터

결과: {광공업생산:D3}는 이제 2023 2/4 분기 데이터를 가리킴 (잘못된 분기)
```

### 3. 구조 변경에 대한 취약성

- **고정된 좌표 의존**: 템플릿이 특정 셀 주소에 하드코딩되어 있어 엑셀 구조가 변경되면 템플릿 전체를 수정해야 함
- **유지보수 어려움**: 엑셀 파일 구조가 변경될 때마다 모든 템플릿의 마커를 찾아 수정해야 함
- **데이터 정합성 위험**: 잘못된 좌표가 헤더나 다른 데이터를 가리킬 위험

## 권장 개선 방안

### 1. 헤더 기반 추출으로 전환 (권장)

현재 `FlexibleMapper`가 이미 구현되어 있으므로, 이를 확장하여:

- 헤더명과 행 값을 조합한 조회: `{시트명:헤더명:행값}`
  - 예: `{광공업생산:2023_3_4:서울}` → "광공업생산" 시트에서 "2023 3/4" 헤더 열과 "서울" 행의 교차점 찾기
- 상대적 위치 지정: `{시트명:헤더명:헤더행+N행}`
  - 예: `{광공업생산:2023_3_4:헤더+2행}` → 헤더 다음 2번째 행

### 2. 하이브리드 방식 (점진적 전환)

절대 좌표와 헤더 기반을 모두 지원하되:

- 새 템플릿은 헤더 기반으로 작성
- 기존 템플릿은 절대 좌표 기반 유지 (하위 호환성)
- 마이그레이션 도구 제공

### 3. 템플릿 검증 도구

- 엑셀 파일 구조 변경 감지
- 템플릿 마커의 유효성 검증
- 사용자에게 변경 사항 알림

## 결론

현재 시스템은 **절대 좌표 기반**으로 작동하며, 데이터 행/열 추가/삭제 시 **심각한 데이터 정합성 문제**가 발생할 수 있습니다. 헤더 기반 추출 방식으로 전환하거나, 최소한 하이브리드 방식을 도입하는 것이 권장됩니다.
