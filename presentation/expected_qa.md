# 예상 Q&A 리스트

## 기술 관련 질문

### Q1: 스키마 기반 시스템의 장점은 무엇인가요?
**핵심 답변:**
- 새로운 보고서 추가 시 코드 수정 없이 스키마만 작성
- 유지보수성 향상
- 데이터 매핑 명확화

**자연스러운 표현:**
"스키마 기반 시스템의 가장 큰 장점은 확장성입니다. 새로운 보고서를 추가할 때 코드를 수정할 필요 없이 JSON 스키마 파일만 작성하면 되거든요. 이렇게 하면 유지보수가 훨씬 쉬워지고, 데이터 매핑도 명확해집니다."

---

### Q2: 가중치 기반 순위 시스템은 어떻게 작동하나요?
**핵심 답변:**
- 17개 시도의 경제지표 분석
- 증감률에 가중치 적용
- 실시간 순위 계산 및 하이라이트

**자연스러운 표현:**
"각 시도의 경제지표 증감률을 분석해서, 중요도에 따라 가중치를 부여합니다. 예를 들어 광공업생산이 서비스업생산보다 더 중요하다면 가중치를 높게 설정하는 거죠. 이렇게 해서 실시간으로 순위를 계산하고 상위/하위 지역을 자동으로 하이라이트합니다."

---

### Q3: 결측치 처리 시스템의 동작 방식을 설명해주세요.
**핵심 답변:**
- 노란색 형광펜으로 시각화
- contenteditable 필드로 직접 수정
- 모달창으로 한눈에 확인

**자연스러운 표현:**
"데이터가 누락되거나 비공개인 경우, 노란색 형광펜으로 표시합니다. 사용자는 그 부분을 클릭해서 직접 수정할 수 있어요. 그리고 모달창에서 모든 결측치를 한눈에 확인하고 일괄 처리할 수도 있습니다."

---

### Q4: 50개 보고서를 어떻게 관리하나요?
**핵심 답변:**
- 카테고리별 분류 (요약/부문별/시도별/통계표)
- 각 보고서마다 독립적인 스키마
- 템플릿 시스템으로 일관성 유지

**자연스러운 표현:**
"50개 보고서를 4개 카테고리로 분류해서 관리합니다. 각 보고서는 독립적인 JSON 스키마를 가지고 있어서, 하나를 수정해도 다른 것에 영향을 주지 않아요. 그리고 템플릿 시스템으로 일관된 포맷을 유지합니다."

---

## 프로젝트 관련 질문

### Q5: 2주라는 짧은 기간에 어떻게 완성했나요?
**핵심 답변:**
- 단계별 개발 (Phase 1-7)
- 스키마 기반 아키텍처로 확장성 확보
- 모듈화로 병렬 개발 가능

**자연스러운 표현:**
"7단계로 나눠서 체계적으로 개발했습니다. 특히 스키마 기반 아키텍처를 도입해서, 초반에 구조를 잘 잡아두니 나중에 보고서를 추가하는 게 훨씬 빨랐어요. 그리고 모듈화된 구조 덕분에 여러 기능을 동시에 개발할 수 있었습니다."

---

### Q6: 가장 어려웠던 부분은 무엇인가요?
**핵심 답변:**
- 의미 기반 마커 시스템 도입
- 다양한 엑셀 시트 형식 처리
- 결측치 처리 로직

**자연스러운 표현:**
"가장 어려웠던 건 의미 기반 마커 시스템이었어요. 처음엔 위치 기반으로 했는데, 엑셀 시트 형식이 조금만 바뀌어도 오류가 발생했거든요. 그래서 데이터의 의미를 파악해서 매핑하는 방식으로 전환했고, 이게 훨씬 안정적이었습니다."

---

### Q7: 실제 사용자 테스트는 했나요?
**핵심 답변:**
- 정답 데이터와 수치 일치 검증
- 다양한 시트 형식 테스트
- 국가데이터처 요구사항 반영

**자연스러운 표현:**
"네, 정답 데이터와 완전히 일치하는지 검증했습니다. 그리고 다양한 엑셀 시트 형식을 테스트해서 안정성을 확보했어요. 국가데이터처의 실제 요구사항도 반영했습니다."

---

## 향후 발전 관련 질문

### Q8: KOSIS API 연동은 어떻게 계획하고 있나요?
**핵심 답변:**
- 실시간 통계 데이터 자동 수집
- 기초자료 수집표 자동 생성
- 우선순위 높음

**자연스러운 표현:**
"KOSIS API를 연동하면 실시간으로 통계 데이터를 가져와서, 기초자료 수집표를 자동으로 생성할 수 있어요. 이게 완성되면 사용자가 엑셀 파일을 준비할 필요 없이, 시스템이 알아서 데이터를 수집하고 보고서를 생성할 수 있습니다. 향후 개선 과제 중 우선순위가 가장 높은 부분입니다."

---

### Q9: 다른 기관에도 적용 가능한가요?
**핵심 답변:**
- 스키마 기반으로 확장 가능
- 템플릿 커스터마이징
- 범용적 아키텍처

**자연스러운 표현:**
"네, 가능합니다. 스키마 기반 아키텍처 덕분에 다른 기관의 보고서 형식에도 적용할 수 있어요. 각 기관의 요구사항에 맞게 스키마와 템플릿만 수정하면 되거든요. 범용적으로 설계했습니다."

---

### Q10: 성능 최적화는 어떻게 했나요?
**핵심 답변:**
- 캐싱을 통한 성능 최적화
- 모듈화로 효율적 처리
- 2주 개발 기간 고려

**자연스러운 표현:**
"캐싱을 도입해서 반복적인 데이터 처리를 최소화했습니다. 그리고 모듈화된 구조로 필요한 부분만 처리하도록 했어요. 2주라는 짧은 기간이었지만, 핵심 기능은 모두 최적화했습니다."

---

## 대응 전략

### 모르는 질문이 나올 때
1. 솔직하게 인정
2. 관련된 아는 부분 설명
3. "추가로 조사해서 답변드리겠습니다" 제안

**예시:**
"좋은 질문이네요. 그 부분은 아직 구체적으로 검토하지 않았지만, 현재 시스템의 스키마 기반 아키텍처를 활용하면 충분히 가능할 것 같습니다. 발표 후에 더 자세히 연구해서 답변드리겠습니다."

---

### 기술적 디테일 질문
- 핵심 개념만 설명
- 너무 깊이 들어가지 않기
- "구현 세부사항은 코드에서 확인 가능합니다" 제안

---

### 비판적 질문
- 감정적으로 대응하지 않기
- 건설적인 피드백으로 받아들이기
- 개선 가능성 언급

**예시:**
"지적해주신 부분은 향후 개선 과제에 포함시켜두었습니다. 특히 KOSIS API 연동을 통해 그 문제를 해결할 수 있을 것 같아요."
