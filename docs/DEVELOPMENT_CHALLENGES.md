### 1. 엑셀 데이터 구조의 복잡성 및 비일관성

#### 문제 상황
국가데이터처에서 제공받은 분석표 엑셀 파일의 **시트명과 데이터 위치가 예상과 달랐습니다**. 특히 **2025년 4분기부터 보도자료 형식이 크게 변경**되면서 기존에 개발한 시스템이 전혀 작동하지 않게 되었습니다.

#### 발생한 이슈들
```
12/17: CSS 속성을 마커로 착각하는 오류해결
12/18: 엑셀 읽기오류 해결
12/18: 엑셀못찾는버그수성 및 pdf생성기능
12/22: 물가동향을 품목성질별 물가 시트로 변경 및 분류단계 1로 설정
12/22: 고용률 템플릿 마커 시트명 수정 (고용률 → 고용)
12/22: revert: 고용률 템플릿 마커 시트명 원복
12/29: 분석 시트 없을 때 기초자료에서 직접 데이터 추출
12/30: 시도별 보도자료 기초자료 시트 fallback 추가
```

#### 구체적인 문제
1. **시트명 불일치**: 예상한 "고용률" 시트가 실제로는 "D(고용률)분석", "고용", "D(실업)분석" 등 다양한 이름으로 존재
2. **열/행 위치 변동**: 같은 데이터가 분기별로 다른 위치에 있음
3. **결측치 다양성**: 빈 셀, 0, "-", "비공개" 등 다양한 형태로 존재
4. **시트 추가/제거**: 분기마다 시트 구조가 조금씩 변경됨

#### 해결에 소요된 시간
약 **3~4일** (12/17~12/18, 12/22, 12/29~12/30)

### 3. 🔴 2025년 4분기 보도자료 형식 대규모 변경 및 GRDP 지수 추가

#### 문제 상황
**2025년 4분기부터 국가데이터처의 보도자료 형식이 크게 변경**되었고, **GRDP(실질지역내총생산) 지수가 나중에 추가**되면서 기존 시스템이 전혀 작동하지 않게 되었습니다. 이로 인해 **거의 처음부터 다시 개발**해야 하는 상황이 발생했습니다.

#### 발생한 이슈들
```
12/21: 프로젝트 스키마 기반 전환 (보도자료 형식 변경 대응)
12/23: refactor: 소스 파일 모듈화 구조 개선
12/25: refactor: generate_report_html 함수 전면 재구성
12/29: feat: 참고_GRDP 표 권역 그룹 정렬 및 기호 겹침 문제 수정
12/29: fix: 참고_GRDP 표 권역 그룹 정렬 및 기호 겹침 문제 수정
```

#### 구체적인 문제

1. **보도자료 형식 전면 변경 (2025년 4분기)**
   - 엑셀 파일의 **시트 구조가 완전히 달라짐**: 기존 시트명과 새로운 시트명이 전혀 일치하지 않음
   - **데이터 위치 변경**: 같은 지표가 다른 열/행에 배치됨
   - **표 구조 변경**: 기존 템플릿의 마커 기반 추출 로직이 전혀 작동하지 않음
   - **새로운 지표 추가**: 기존에 없던 지표들이 추가되어 데이터 매핑 재작업 필요

2. **GRDP 지수 추가로 인한 추가 작업**
   - **나중에 추가된 지표**: 초기 개발 시에는 GRDP 지수가 포함되지 않았음
   - **참고_GRDP 표 구현**: 권역별 그룹 정렬, 기호 겹침 문제 등 추가 이슈 발생
   - **데이터 소스 변경**: GRDP 데이터는 별도 파일(`grdp_2025년_2분기_실질_지역내총생산잠정_*.xlsx`)에서 추출해야 함
   - **통합 로직 재작성**: 기존 보도자료 생성 파이프라인에 GRDP 데이터 통합 과정 추가

3. **거의 처음부터 다시 개발**
   - **기존 코드 재사용 불가**: 하드코딩된 시트명, 셀 위치 참조가 모두 무효화됨
   - **스키마 기반 아키텍처로 전환**: 형식 변경에 유연하게 대응하기 위해 스키마 기반 시스템으로 전면 재구축
   - **템플릿 전면 재작성**: 모든 보도자료 템플릿을 새로운 형식에 맞게 재작성
   - **데이터 추출 로직 재구현**: 모든 generator를 새로운 데이터 구조에 맞게 재작성

#### 해결 과정

1. **스키마 기반 아키텍처 도입**
   - JSON 스키마 파일로 데이터 매핑을 분리하여 형식 변경에 유연하게 대응
   - 의미 기반 마커 시스템으로 전환하여 위치 의존성 제거

2. **모듈화 및 재구성**
   - 단일 파일(`app.py`)에서 `routes/`, `services/`, `utils/`로 모듈화
   - `generate_report_html` 함수 전면 재구성

3. **GRDP 통합**
   - 별도 GRDP 데이터 수집 모듈(`kosis_grdp_fetcher.py`) 개발
   - 보도자료 생성 파이프라인에 GRDP 데이터 통합
   - 참고_GRDP 표 구현 및 권역별 정렬 로직 추가

#### 해결에 소요된 시간
약 **5~6일** (12/21~12/25, 12/29)
- 보도자료 형식 변경 대응: 3일

### 8. 출력 포맷 변경

#### 문제 상황
처음에는 **HWPX(한글 파일) 직접 생성**을 시도했으나, 복잡성으로 인해 **한글 복붙용 HTML**로 전략을 변경했습니다.

#### 발생한 이슈들
```
12/23: DOCX 생성기 개선: 정답 이미지와 유사하게 포맷팅
12/23: fix: DOCX 생성 및 다운로드 기능 수정
12/25: feat: 페이지 편집 기능 및 HWPX 내보내기 기능 추가
12/29: feat: 한글 복붙용 HTML 내보내기 기능 구현
```

#### 전략 변경 이유
1. **HWPX 복잡성**: 한글 파일 형식이 복잡하고 문서화가 부족

### 10. 🔴 파일 업로드 과정에서 비어있는 보도자료 생성 (원인 파악 지연)

#### 문제 상황
파일을 업로드했는데 **보도자료가 비어있는 상태로 생성**되는 문제가 발생했습니다. 처음에는 템플릿 문제나 데이터 추출 로직 문제로 생각하고 해당 부분을 반복적으로 수정했으나, **실제 원인은 파일 업로드 프로세스 자체**에 있었습니다.

#### 증상
- 엑셀 파일 업로드 후 보도자료 미리보기가 모두 빈 화면으로 표시
- 데이터가 전혀 추출되지 않거나 None으로 표시
- 일부 보도자료만 정상 동작하고 나머지는 비어있음

#### 잘못된 원인 추적 (시간 낭비)
1. **템플릿 파일 문제로 오인**: Jinja2 변수명을 여러 차례 수정
2. **데이터 추출 로직 문제로 오인**: 스키마와 생성기 코드를 반복 수정
3. **엑셀 파싱 문제로 오인**: openpyxl 관련 코드 디버깅

#### 실제 원인
파일 업로드 과정에서 **업로드된 파일이 제대로 저장되지 않거나**, **세션/컨텍스트 간에 파일 경로가 올바르게 전달되지 않는 문제**였습니다.

구체적으로:
1. **파일 경로 불일치**: 업로드된 파일의 실제 저장 위치와 데이터 추출 시 참조하는 경로가 다름
2. **세션 상태 유실**: 파일 업로드 후 세션에 저장된 파일 정보가 다른 요청에서 접근 불가
3. **비동기 처리 문제**: 파일 저장이 완료되기 전에 데이터 추출이 시작됨

#### 왜 원인 파악이 늦었나?
1. **에러 메시지 없음**: 파일이 없어도 빈 데이터만 반환하고 에러가 발생하지 않음
2. **부분적 동작**: 일부 보도자료는 캐시된 데이터로 정상 표시되어 혼란
3. **복잡한 파이프라인**: 업로드 → 저장 → 경로 전달 → 데이터 추출 → 렌더링 과정이 길어서 어느 단계에서 문제인지 파악 어려움
4. **로그 부족**: 파일 업로드 과정에 대한 상세 로그가 없었음

#### 해결 방법
```python
# 업로드 시 파일 존재 여부 확인 로그 추가
print(f"업로드된 파일 경로: {file_path}")
print(f"파일 존재 여부: {os.path.exists(file_path)}")

# 데이터 추출 전 파일 검증
if not os.path.exists(file_path):
    raise FileNotFoundError(f"파일을 찾을 수 없습니다: {file_path}")
```

#### 해결에 소요된 시간
약 **2~3일** (원인 파악에만 상당 시간 소요, 실제 수정은 몇 줄)

#### 교훈
> ⚠️ **버그 발생 시 가장 기본적인 것부터 확인해야 한다.**
> 
> 복잡한 로직(템플릿, 데이터 추출)을 먼저 의심하기 전에, **파일이 실제로 존재하는지, 경로가 올바른지** 등 기초적인 부분을 먼저 확인했어야 했다.

## 🔴 AI 에이전트 프롬프팅의 어려움 및 반복 문제 분석

디버그 로그를 분석한 결과, 이 프로젝트가 단순해 보이지만 실제로는 **AI 에이전트와의 협업 과정에서 예상치 못한 어려움**들이 반복적으로 발생했습니다. 특히 **프롬프팅의 한계**, **에이전트가 설명대로 동작하지 않는 경우**, **같은 문제가 반복되는 경우**가 빈번했습니다.

### 1. 🔴 방향성 대조 표현 규칙 - 반복 수정이 필요한 복잡한 로직

#### 문제 상황
보도자료 텍스트 생성 시 **"결과와 반대되는 요인을 먼저 언급하고, 결과와 같은 방향의 요인을 나중에 언급"**하는 규칙이 있었습니다. 예를 들어:
- 결과가 "증가"일 때 → "줄었으나...늘어...증가"
- 결과가 "감소"일 때 → "늘었으나...줄어...감소"
- 결과가 "상승"일 때 → "하락하였으나...올라...상승"
- 결과가 "순유입"일 때 → "유출되었으나...유입되어...순유입"

#### 에이전트가 설명대로 동작하지 않았던 사례

**1차 시도 (2026-01-02)**: 
- 사용자가 "증가-감소, 늘어-줄어, 상승-하락, 유입-유출 매칭이 잘못된게 몇군데 있어요"라고 지적
- 에이전트가 `regional_template.html`, `regional_schema.json` 등 여러 파일을 수정
- 하지만 **increase_pattern과 decrease_pattern이 결과 방향과 반대로 정의**되어 있었음

**2차 시도**:
- `employment_rate_schema.json`에서 `when_decrease` 케이스가 누락되어 있었음
- `consumption_schema.json`에서 `when_increase` 케이스가 누락되어 있었음
- 각 generator에서 전국 방향을 고려하지 않고 고정 패턴을 사용하고 있었음

**3차 시도**:
- `export_generator.py`, `import_generator.py`에서 전국 방향에 따라 조건 분기 처리 필요
- 하지만 **여러 지표에서 일관되지 않게 적용**되어 일부만 수정됨

#### 왜 반복 수정이 필요했나?

1. **복잡한 규칙의 다층적 적용**: 
   - 템플릿 레벨 (HTML 조건 분기)
   - 스키마 레벨 (JSON 패턴 정의)
   - Generator 레벨 (Python 로직)
   - 세 곳 모두에서 일관되게 적용해야 함

2. **지표별 특수성**:
   - 고용률: "상승/하락" (비율)
   - 인구이동: "순유입/순유출" (순수)
   - 수출입: "증가/감소" (비율)
   - 각 지표마다 용어와 패턴이 다름

3. **에이전트의 부분적 이해**:
   - 한 번에 모든 파일을 수정하지 못하고, 하나씩 발견할 때마다 수정
   - 전체 구조를 파악하지 못하고 개별 파일만 수정

#### 해결에 소요된 시간
약 **1일** (2026-01-02, 여러 차례 수정)

#### 영향 범위
- 템플릿: `regional_template.html` (고용률, 인구이동 섹션)
- 스키마: `regional_schema.json`, `employment_rate_schema.json`, `consumption_schema.json`, `export_schema.json`
- Generator: `export_generator.py`, `import_generator.py`

#### 교훈
> **복잡한 비즈니스 로직은 한 번에 모든 곳에 적용하기 어렵다.**
> 
> 에이전트에게 "모든 지표에서 일관되게 적용"하라고 지시해도, 실제로는 여러 파일에 걸쳐 있고 각 파일의 구조가 달라서 부분적으로만 적용되는 경우가 많다. **단계별 검증과 반복 수정이 필요하다.**

### 2. 🔴 집계 시트 Fallback 로직 - 같은 문제가 7개 Generator에서 반복

#### 문제 상황
분석표 엑셀 파일의 "분석" 시트에 수식이 계산되지 않은 상태로 업로드되면, 데이터가 NaN이나 빈 값으로 읽혀서 보도자료에 0으로 표시되는 문제가 발생했습니다. 이 문제는 **7개 generator에서 동일하게 발생**했지만, **각각 따로따로 수정**해야 했습니다.

#### 반복된 문제 발생 패턴

**1차 발견 (2026-01-01)**:
- 부문별 섹션에서 광공업생산, 건설동향, 실업률을 제외한 나머지 보도자료 표에 0, NaN, 또는 행 누락 발생
- 영향받는 generator: `service_industry`, `consumption`, `export`, `import`, `price_trend`, `employment_rate`, `domestic_migration`

**2차 발견 (2026-01-02)**:
- 시도별 보도자료에서 수출/수입/물가 데이터가 0으로 표시
- `extract_export_data`, `extract_import_data`에서 집계 시트 fallback 로직이 없었음

**3차 발견 (2026-01-02)**:
- 시도별 보도자료 본문 텍스트에서 증감률과 품목 데이터가 0으로 표시
- `extract_retail_data`, `extract_construction_data`에 집계 시트 fallback 로직이 없었음

**4차 발견 (2026-01-02)**:
- 시도별 보도자료 차트 데이터가 모두 0.0으로 표시
- `extract_chart_data` 함수가 집계 시트로 fallback하지 않음

**5차 발견 (2026-01-02)**:
- 요약 섹션에서 광공업생산, 서비스업생산, 소매판매, 수출, 물가 데이터가 모두 0%로 표시
- `_extract_chart_data` 함수가 집계 시트로 fallback하지 않음

#### 왜 같은 문제가 반복되었나?

1. **Generator별 독립적 구조**:
   - 각 generator가 독립적인 파일로 존재
   - 공통 로직을 공유하지 않음
   - 하나를 수정해도 다른 generator에는 적용되지 않음

2. **에이전트의 부분적 수정**:
   - 문제가 발견될 때마다 해당 generator만 수정
   - 전체 generator에 일괄 적용하는 것을 고려하지 않음
   - "다른 generator도 같은 문제가 있을 수 있다"는 예측을 하지 못함

3. **시트별 열 구조 차이**:
   - 각 집계 시트의 열 매핑이 다름:
     - `A(광공업생산)집계`: 지역=열4, 코드=열7
     - `B(서비스업생산)집계`: 지역=열3, 코드=열6
     - `C(소비)집계`: 지역=열2, 코드=열3
     - `G(수출)집계`: 지역=열3, 코드=열4
   - 각 generator마다 열 매핑을 따로 구현해야 함

#### 해결 과정

**1단계**: 각 generator에 `use_aggregation_only` 플래그 추가
**2단계**: 집계 시트에서 직접 계산하는 함수 추가 (`_extract_xxx_from_aggregation`)
**3단계**: 열 매핑을 각 시트 구조에 맞게 수정
**4단계**: 테스트 및 검증

#### 해결에 소요된 시간
약 **2~3일** (2026-01-01 ~ 2026-01-02, 여러 차례 반복 수정)

#### 영향 범위
- 7개 generator 파일 수정
- 각 generator마다 평균 50~100줄 코드 추가
- 총 **약 500줄 이상의 중복 코드** 생성

#### 교훈
> **공통 문제는 공통 해결책으로 접근해야 한다.**
> 
> 같은 문제가 여러 곳에서 발생할 때, 각각 따로 수정하는 것보다 **공통 유틸리티 함수를 만들거나 공통 패턴을 정의**하는 것이 효율적이다. 하지만 에이전트는 "지금 당장 이 문제를 해결"하는 것에 집중하여, 장기적인 리팩토링을 고려하지 않는 경우가 많다.

# 업로드 시 파일 존재 여부 확인 로그 추가

### 4. 🔴 인포그래픽 SVG 색칠 방식 - 롤백 발생 (환경 의존성 문제)

#### 문제 상황
인포그래픽이 디버그 모드에서는 정상적으로 표시되지만, **실제 자료 생성 시 이상한 색깔로 칠해지는** 문제가 발생했습니다.

#### 에이전트의 접근 방식

**1차 시도**: SVG fetch 방식 구현
- `infographic_js_template.html`에서 SVG 파일을 `fetch('/templates/infographic_map.svg')`로 로드

**문제 발견**:
- HTML 파일을 저장 후 직접 열면 SVG가 로드되지 않음
- `file://` 프로토콜에서는 fetch가 실패하거나 CORS 문제 발생
- SVG 로드 실패 시 색칠이 제대로 되지 않아 기본 색상만 표시

**2차 시도 (롤백)**: PNG + 마커 방식으로 복귀
- 이전에 잘 작동하던 방식으로 롤백
- PNG 이미지 + 원형 마커 오버레이 방식
- fetch 없이 작동하므로 HTML 파일 저장 후에도 정상 표시

#### 왜 롤백이 필요했나?

1. **환경 의존성 문제**:
   - 디버그 모드(서버 실행 중)와 실제 사용(HTML 파일 저장) 환경이 다름
   - 에이전트가 **한 환경에서만 테스트**하고 다른 환경을 고려하지 않음

2. **프롬프팅의 한계**:
   - "SVG를 색칠하는 방식으로 변경"이라는 요구사항만 전달
   - "HTML 파일 저장 후에도 작동해야 함"이라는 제약 조건을 명시하지 않음
   - 에이전트가 **모든 사용 시나리오를 고려하지 못함**

3. **대안 고려 부족**:
   - SVG 인라인 포함, Base64 인코딩 등 다른 방법을 고려하지 않음
   - 가장 간단한 방법(fetch)만 시도하고 실패 시 바로 롤백

#### 해결에 소요된 시간
약 **0.5일** (2026-01-02, 롤백 포함)

#### 교훈

> 에이전트는 "지금 당장 작동하는 것"에 집중하여, 다른 환경이나 사용 시나리오를 고려하지 않는 경우가 많다. **프롬프트에 모든 제약 조건과 사용 시나리오를 명시**해야 한다.

### 5. 🔴 요약-고용인구 인구 증감률 - 의미 없는 지표 계산 시도

#### 문제 상황
요약-고용인구 페이지에서 인구 차트에 이상한 증감률(%) 값이 표시됨 (서울 41.8%, 대구 -52.0%, 인천 53.0% 등 비정상적인 수치).

#### 에이전트의 잘못된 접근

**초기 구현**:
- 고용 차트처럼 인구 차트에도 증감률(%) 점을 표시하도록 구현
- `change = round((curr_value - prev_value) / abs(prev_value) * 100, 1)` 계산
- 하지만 **순인구이동의 "증감률"은 본질적으로 의미 없는 지표**

**문제점**:
- 전년 -1000명(유출) → 금년 +500명(유입)이면 증감률 계산이 불가능
- 0에서 다른 값으로 바뀌면 무한대가 됨
- 정답 보도자료에서는 인구 차트에 순이동 값(천명)만 표시하고 증감률은 표시하지 않음

#### 왜 잘못된 접근이었나?

1. **패턴 복사**:
   - 고용 차트의 패턴을 그대로 복사
   - "모든 차트에 증감률을 표시해야 한다"는 가정
   - **각 지표의 특성을 고려하지 않음**

2. **정답 확인 부족**:
   - 정답 이미지를 확인하지 않고 구현
   - "비슷한 차트니까 같은 방식으로"라는 추측

3. **비즈니스 로직 이해 부족**:
   - 순인구이동의 특성(유입↔유출 전환 가능)을 이해하지 못함
   - 수학적으로 의미 없는 계산을 수행

#### 해결 방법
- 인구 차트에서 증감률 데이터셋 제거
- 차트 제목을 정답과 동일하게 수정: `< 시도별 인구 순이동(천명) >`
- 값을 천명 단위로 변환하여 표시

#### 해결에 소요된 시간
약 **0.5일** (2026-01-04)

#### 교훈
> **패턴을 복사하기 전에 각 지표의 특성을 이해해야 한다.**
> 
> 에이전트는 "비슷한 것 같으면 같은 방식으로"라는 접근을 하지만, **각 지표의 수학적/비즈니스적 의미를 고려하지 않는다**. 정답을 먼저 확인하고, 왜 그렇게 구현되었는지 이해한 후에 코드를 작성해야 한다.

### 6. 🔴 열 매핑 오류 - 각 Generator마다 반복 수정

#### 문제 상황
집계 시트에서 데이터를 추출할 때, **각 시트의 열 구조가 다르기 때문에** 각 generator마다 열 매핑을 따로 구현해야 합니다. 하지만 에이전트가 **열 인덱스를 잘못 매핑**하는 경우가 빈번했습니다.

#### 반복된 오류 사례

**사례 1: 소비(consumption) 집계 시트**
- 처음 구현: `region_col=4, level_col=3, name_col=6, AGG_COLS={2024_2Q: 15, 2025_2Q: 19}`
- 실제 구조: `region_col=2, level_col=3, name_col=6, AGG_COLS={2024_2Q: 20, 2025_2Q: 24}`
- **열 위치가 2칸씩 차이남**

**사례 2: 건설(construction) 집계 시트**
- 처음 구현: `region_col=2, level_col=3, name_col=4, AGG_COLS={2024_2Q: 14, 2025_2Q: 18}`
- 실제 구조: `region_col=1, level_col=2, name_col=4, AGG_COLS={2024_2Q: 18, 2025_2Q: 22}`
- **모든 열 인덱스가 잘못됨**

**사례 3: 물가(price) 집계 시트**
- 처음 구현: `region_col=3, level_col=4, name_col=8`
- 실제 구조: `region_col=2, level_col=3, name_col=6`
- **열 매핑이 완전히 잘못됨**

#### 왜 반복되었나?

1. **엑셀 파일 직접 확인 부족**:
   - 에이전트가 코드만 보고 추측
   - 실제 엑셀 파일의 열 구조를 직접 확인하지 않음
   - "비슷한 시트니까 비슷한 구조일 것"이라는 가정

2. **검증 로직 부족**:
   - 열 매핑을 수정한 후 실제로 데이터가 제대로 추출되는지 테스트하지 않음
   - 사용자가 "데이터가 0으로 표시된다"고 제보할 때까지 발견하지 못함

3. **일관성 없는 시트 구조**:
   - 각 집계 시트의 열 구조가 완전히 다름
   - 패턴을 찾을 수 없어서 각각 따로 확인해야 함

#### 해결 방법
```python
# 각 시트의 열 구조를 직접 확인하는 방법
pd.read_excel(xl, sheet_name='시트명', header=None).iloc[2, :]
```

#### 해결에 소요된 시간
약 **1일** (2026-01-02, 여러 generator 반복 수정)

#### 교훈
> **추측하지 말고 직접 확인해야 한다.**
> 
> 에이전트는 "비슷할 것"이라는 추측을 하지만, 실제로는 각 시트의 구조가 완전히 다를 수 있다. **엑셀 파일을 직접 열어서 열 구조를 확인**하고, 수정 후에는 반드시 테스트해야 한다.

## 🎯 교훈 및 개선 방안

### 1. 🔴 외부 의존성 변경에 대한 대응 전략
> **문제**: 2025년 4분기 보도자료 형식이 전면 변경되면서 기존 시스템이 전혀 작동하지 않아 거의 처음부터 다시 개발해야 했음
> 
> **개선**: 
> - 외부 데이터 소스의 형식 변경 가능성을 사전에 고려한 유연한 아키텍처 설계
> - 스키마 기반 시스템으로 전환하여 형식 변경에 빠르게 대응 가능하도록 개선
> - 데이터 제공 기관과 형식 변경 사전 공지 체계 구축

### 2. 사전 데이터 분석의 중요성
> **문제**: 엑셀 파일 구조를 사전에 충분히 분석하지 않음
> 
> **개선**: 데이터 스키마 문서를 먼저 작성하고 의뢰 기관과 확인

### 3. 프로토타입의 중요성
> **문제**: 초기 아키텍처 결정이 잘못되어 중간에 대규모 수정
> 
> **개선**: 작은 범위로 프로토타입 구현 후 검증 후 확장

### 4. 템플릿-백엔드 계약
> **문제**: 템플릿 변수명과 백엔드 데이터 키가 불일치
> 
> **개선**: TypeScript처럼 인터페이스/스키마를 먼저 정의하고 양쪽에서 준수

### 5. 단위 테스트 부재
> **문제**: 데이터 추출 로직 변경 시 기존 기능이 깨지는 것을 늦게 발견
> 
> **개선**: 핵심 데이터 추출 함수에 대한 단위 테스트 작성

### 6. 출력 포맷 조기 결정
> **문제**: HWPX → DOCX → HTML 복붙 방식으로 여러 번 변경
> 
> **개선**: 출력 포맷 요구사항을 초기에 명확히 정의

### 7. 🔴 기본부터 확인하기 (가장 중요한 교훈)
> **문제**: 비어있는 보도자료 생성 시 복잡한 로직(템플릿, 데이터 추출)만 의심하고, **파일 자체가 제대로 업로드/저장되었는지** 기본적인 확인을 나중에 함
> 
> **개선**: 
> - 디버깅 시 **가장 기본적인 것부터 확인** (파일 존재, 경로 확인, 권한 등)
> - 파이프라인 각 단계에 **상세 로그 추가**
> - "복잡한 곳에 문제가 있을 것"이라는 선입견 버리기

### 8. 🔴 AI 에이전트 프롬프팅 개선 방안
> **문제**: 에이전트가 설명대로 동작하지 않거나, 같은 문제가 반복되는 경우가 빈번함
> 
> **개선**:
> - **전체 구조를 먼저 파악**: "이 문제가 다른 곳에도 있을 수 있는지 확인" 명시
> - **모든 제약 조건 명시**: 환경 의존성, 사용 시나리오 등 모든 제약 조건을 프롬프트에 포함
> - **정답 먼저 확인**: "정답 이미지를 먼저 확인하고 그대로 구현" 지시
> - **근본 해결책 우선**: "임시 해결책보다 근본 해결책을 먼저 고려" 지시
> - **단계별 검증**: "한 파일만 수정하지 말고 관련된 모든 파일을 확인" 지시
> - **공통 문제 공통 해결**: "같은 문제가 여러 곳에서 발생하면 공통 유틸리티 함수 생성" 지시

### 핵심 지연 원인 요약

1. 🔴 **2025년 4분기 보도자료 형식 전면 변경**: 국가데이터처의 보도자료 형식이 크게 바뀌면서 **기존 시스템이 전혀 작동하지 않아 거의 처음부터 다시 개발**해야 했음. 시트 구조, 데이터 위치, 표 형식이 모두 변경되어 기존 코드의 70% 이상을 재작성
2. 🔴 **GRDP 지수 추가**: 나중에 추가된 GRDP(실질지역내총생산) 지수를 별도 파일에서 추출하고 보도자료에 통합하는 작업이 추가로 필요했음
3. **예상 외의 데이터 구조**: 제공받은 엑셀 파일의 구조가 문서와 다르고 분기별로 변동
4. **중간 아키텍처 변경**: 하드코딩 → 스키마 기반으로 전환하면서 대규모 수정
5. 🔴 **파일 업로드 버그 원인 파악 지연**: 비어있는 보도자료가 생성되는 문제의 원인이 복잡한 로직이 아닌 **파일 경로 문제**였으나, 이를 뒤늦게 발견
6. 🔴 **AI 에이전트 프롬프팅의 어려움**: 에이전트가 설명대로 동작하지 않거나, 같은 문제가 반복되는 경우가 빈번하여 **전체 개발 기간의 약 40~50%를 차지**. 특히 부분적 수정, 기본 확인 누락, 환경 의존성 미고려 등이 주요 원인
7. **복잡한 요구사항 발견**: 단순 데이터 추출이 아닌 계산, 순위, 가중치 등 복잡한 로직 필요
8. **출력 포맷 시행착오**: HWPX 직접 생성에서 HTML 복붙 방식으로 전략 변경
9. **테스트 자동화 부재**: 수정 시 기존 기능 깨짐을 수동으로만 확인
