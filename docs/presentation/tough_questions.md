## 3. 기술적 도발 질문

### Q: "Flask 말고 Django나 FastAPI 쓰지, 왜 Flask야?"
**상황:** 기술 선택에 대한 비판

**대응:**
> "Flask는 가볍고 **의존성이 최소화**되어서 국가데이터처 내부망 환경에 적합합니다. 외부 프로그램 반입이 제한적인데, Flask는 pip 패키지 몇 개만 설치하면 바로 동작합니다. Django는 ORM, Admin 등 불필요한 의존성이 많고, FastAPI의 async는 분기별 보도자료 생성에 오버스펙입니다."

---

### Q: "왜 데이터베이스 안 썼어? JSON 파일로 하면 느리지 않아?"
**상황:** DB 미사용에 대한 지적

**대응:**
> "국가데이터처 내부망에서 DB 서버를 별도로 운영하기 어렵습니다. 이 시스템은 분기별 보도자료 생성이 목적이라 실시간 대용량 처리가 필요 없습니다. JSON 파일 기반이면 **엑셀처럼 담당자가 직접 수정**할 수 있고, 추가 서버 설치 없이 윈도우 PC에서 바로 실행 가능합니다."

---

### Q: "이거 그냥 엑셀 매크로로 하면 안 되나?"
**상황:** 왜 웹 시스템으로 만들었냐는 질문

**대응:**
> "엑셀 매크로도 고려했지만, 50개 보도자료를 일괄 관리하고 결측치를 시각적으로 표시하려면 한계가 있습니다. 또한 매크로는 유지보수가 어렵고, 보안 설정에 따라 실행이 차단될 수 있습니다. 웹 시스템은 **브라우저만 있으면 동작**하고, 담당자가 변경되어도 인수인계가 쉽습니다."

---

### Q: "한글(HWP) 직접 출력 왜 안 해?"
**상황:** HWP 생성 미지원 지적

**대응:**
> "HWP는 한컴오피스 **독점 포맷**이라 공식 라이브러리가 없습니다. 비공식 라이브러리(pyhwp 등)도 있지만 기능이 제한적이고, 국가데이터처 내부망에서 외부 패키지 설치 승인받기도 어렵습니다. 그래서 **'한글 복붙용 HTML'**을 제공합니다. 담당자분들이 HTML을 열고 Ctrl+A, Ctrl+C 후 한글 문서에 Ctrl+V 하면 서식이 유지된 채 붙여넣기 됩니다. 실무에서도 이 방식이 가장 안정적이라고 피드백 받았습니다."

**추가 질문: "그럼 서식이 깨지지 않아?"**
> "HTML을 한글에서 지원하는 형식으로 최적화했습니다. 표, 글꼴, 줄간격 등 기본 서식은 유지됩니다. 일부 미세 조정은 필요할 수 있지만, 처음부터 작성하는 것보다 훨씬 빠릅니다."

---

### Q: "왜 PDF가 아니라 HTML로 생성해?"
**상황:** 출력 포맷 선택 이유 질문

**대응:**
> "최종 산출물이 **한글(HWP) 문서**이기 때문입니다. PDF는 편집이 안 되고, 한글 문서로 변환도 어렵습니다. HTML은 **복사-붙여넣기로 한글에 바로 옮길 수 있고**, 결측치 수정 등 편집도 가능합니다. 국가데이터처 실무에서는 최종 보도자료를 한글로 제출하기 때문에 이 방식이 가장 현실적입니다."

**추가 질문: "웹에서 바로 PDF 변환하면 되잖아"**
> "PDF 변환 라이브러리(wkhtmltopdf, weasyprint 등)를 설치해야 하는데, 내부망에서 외부 프로그램 설치 승인이 필요합니다. 브라우저 인쇄(Ctrl+P → PDF 저장)로 간단히 PDF 생성도 가능하지만, 어차피 최종본은 한글이라 HTML이 더 유용합니다."

## 7. 🤖 자연어처리(NLP) 관련 질문

### Q: "자연어처리 기술은 안 썼어? 왜?"
**상황:** 요즘 LLM/NLP가 핫한데 왜 안 썼냐는 질문

**대응:**
> "핵심적인 이유는 **국가데이터처의 내부망 환경**입니다. 인터넷 접근이 제한되어 있어서 OpenAI API, Claude API 같은 외부 LLM 서비스를 사용할 수 없습니다. 로컬 LLM을 설치하려면 고사양 GPU가 필요한데, 업무용 윈도우 PC에서는 어렵습니다."

**추가 설명:**
> "또한 이 보도자료는 **정형화된 통계 데이터**입니다. 자연어처리는 비정형 텍스트 분석에 강점이 있는데, 이 시스템은 '숫자 → 문장 변환'이 핵심이라 규칙 기반 템플릿이 더 적합합니다. 예를 들어 '증가율 2.1%'를 '전년동분기대비 2.1% 증가'로 변환하는 건 if-else 로직으로 충분합니다."

---

### Q: "GPT로 보도자료 문장 생성하면 더 자연스럽지 않아?"
**상황:** LLM 활용 가능성 질문

**대응:**
> "자연스러움 측면에서는 맞습니다. 하지만 세 가지 이유로 사용하지 않았습니다:
> 1. **내부망 제한**: 국가데이터처는 인터넷 접근이 제한되어 외부 API 호출 불가
> 2. **정확성 보장**: 통계 보도자료는 숫자 하나 틀리면 안 됩니다. LLM은 할루시네이션 위험이 있음
> 3. **재현 가능성**: 같은 입력에 같은 출력이 나와야 하는데, LLM은 매번 다른 결과 생성
>
> 공문서 특성상 '자연스러움'보다 '정확성과 일관성'이 더 중요합니다."

---

### Q: "그럼 로컬에서 돌아가는 오픈소스 LLM은?"
**상황:** Llama, Mistral 등 로컬 LLM 제안

**대응:**
> "좋은 제안이지만, 현실적인 제약이 있습니다:
> 1. **하드웨어 제한**: 로컬 LLM은 최소 16GB VRAM GPU 필요, 업무용 PC는 내장 그래픽
> 2. **프로그램 반입 제한**: 국가데이터처는 외부 프로그램 설치에 보안 승인 필요
> 3. **유지보수**: 모델 업데이트, 버전 관리 등 추가 부담
>
> 향후 국가데이터처 인프라가 개선되면 검토해볼 수 있지만, 현재 환경에서는 **규칙 기반 템플릿이 가장 현실적**입니다."

---

### Q: "텍스트 요약이나 키워드 추출 같은 건?"
**상황:** NLP의 다른 기능 활용 가능성 질문

**대응:**
> "이 프로젝트에서는 **텍스트 생성**이 아니라 **텍스트 매핑**이 핵심입니다. 입력(엑셀)에 있는 데이터를 정해진 형식의 문장으로 변환하는 거죠. 요약이나 키워드 추출이 필요한 비정형 텍스트가 없습니다. 모든 데이터가 정형화된 숫자와 지역명, 업종명이라서 규칙 기반이 더 효율적입니다."

---

### Q: "미래에는 AI 연동 계획 있어?"
**상황:** 향후 발전 방향 질문

**대응:**
> "국가데이터처 인프라가 개선되면 두 가지를 검토해볼 수 있습니다:

## 8. 🔥 코드 구현 방식 비판 (심화 기술 질문)

### Q: "스키마 기반이라면서 왜 매핑 정보가 코드에 하드코딩되어 있어?"
**상황:** `INDUSTRY_NAME_MAP`, `REGION_GROUPS` 등이 Generator 클래스에 직접 정의되어 있는 것을 지적

**실제 코드 예시:**
```python
# 광공업생산Generator 클래스 내부

    ...
}
```

**대응:**
> "좋은 지적입니다. 이상적으로는 매핑 정보도 JSON 스키마에 분리해야 합니다. 현재 구조에서 스키마 파일은 '데이터 구조 정의'에, Generator는 '데이터 변환 로직'에 집중하도록 역할을 나눴습니다. 매핑 사전은 '변환 로직'의 일부로 봤기 때문에 Generator에 두었는데, 향후 리팩토링 시 분리하는 게 맞습니다."

**추가 설명 (깊이 물어볼 경우):**
> "스키마에 매핑을 넣지 않은 실용적인 이유도 있습니다. 매핑 로직은 if-else나 예외 처리가 필요한 경우가 많아서 JSON으로 표현하기 어렵습니다. 예를 들어 '전자 부품, 컴퓨터...'를 '반도체·전자부품'으로 변환할 때 부분 일치 검사나 우선순위 처리가 필요하거든요."

### Q: "스키마 파일이 있는데 왜 Generator에서 또 데이터 구조를 정의해?"

**대응:**
> "엄격한 MVC는 아니지만 비슷한 구조입니다.  
> - **Model**: Generator가 엑셀에서 데이터 추출 (데이터 계층)  
> - **View**: Jinja2 템플릿이 HTML 렌더링 (표현 계층)  
> - **Controller**: Flask 라우트가 요청 처리 (제어 계층)  
> 
> 추가로 스키마가 '데이터 계약(Data Contract)' 역할을 합니다."

## 🎯 당황스러운 질문 대응 원칙

1. **잠시 멈추기** - 바로 대답하지 말고 1-2초 생각
2. **솔직하게** - 모르면 모른다고, 약점은 인정
3. **전환하기** - 약점 인정 후 강점으로 연결
4. **짧게** - 길게 변명하면 더 의심받음
5. **웃기** - 너무 심각하게 받아들이지 않기

## 🔥 꼬장부리는 심사위원 대응 전략

### 유형 1: 같은 질문 계속 반복 (이해 못 함)

**상황:** 분명히 답변했는데 "그래서 왜 그렇게 했다는 거야?"라고 또 물어봄

**전략: 더 단순하게 + 비유 사용**

```
1차 답변: 기술적 설명
   ↓ (이해 못 함)
2차 답변: 핵심만 한 문장으로
   ↓ (또 이해 못 함)
3차 답변: 일상 비유로 설명
```

**예시:**
> **1차**: "스키마 기반 아키텍처로 데이터 구조를 분리해서..."
> **2차**: "설정 파일만 수정하면 코드 수정 없이 새 보도자료를 추가할 수 있습니다."
> **3차**: "레고 블록처럼 조립하는 방식입니다. 새 블록(보도자료)을 끼우기만 하면 됩니다."

**꿀팁 멘트:**
> "제가 설명이 부족했네요. 쉽게 말씀드리면..."
> "핵심만 말씀드리면..."
> "비유로 설명드리자면..."

---

### 유형 2: 자기 생각이 맞다고 고집 (반박형)

**상황:** "아니 그건 그렇게 하면 안 되지. OOO 방식이 맞아"

**전략: 동의 후 환경 제약 강조**

**틀린 예:**
> ❌ "아니요, 그건 이 경우에 맞지 않습니다" (정면 충돌)

**맞는 예:**
> ✅ "말씀하신 방법도 일반적으로 좋은 방법입니다. 다만 **국가데이터처 내부망 환경**에서는 [제약]이 있어서 이 방식을 선택했습니다."

**구체적 대응:**
| 심사위원 주장 | 대응 |
|--------------|------|
| "DB 써야지" | "맞습니다. 다만 내부망에서 DB 서버 운영이 제한되어..." |
| "React가 낫지" | "동의합니다. 다만 npm/node 설치 승인이 어려워..." |
| "AI 써야지" | "좋은 방향입니다. 다만 인터넷 차단 환경이라..." |
| "클래스로 통일해" | "맞는 지적입니다. 리팩토링 과제로 인지하고 있습니다." |

**핵심 패턴:**
> "**맞습니다** / **좋은 지적입니다** + **다만** [환경 제약/시간 제약] + **그래서** [현재 선택]"

---

### 유형 3: 계속 꼬투리 잡기 (트집형)

**상황:** 뭘 답해도 "그건 그렇고...", "근데 그건...", "그래도..."

**전략: 손절 멘트로 마무리**

**손절 멘트:**
> "지적해주신 부분은 **향후 개선 과제로 포함**시켜두었습니다. 발표 후에 더 고민해보겠습니다."

> "좋은 피드백 감사합니다. **프로토타입 단계**라 완벽하지 않지만, 핵심 기능 구현에 집중했습니다."

> "말씀하신 부분은 **시간 제약**으로 미처 반영하지 못했습니다. 추가 개발 시 반드시 고려하겠습니다."

> "네, 인정합니다. 다음 질문 있으시면 말씀해주세요."

**포인트:** 
- 3번 이상 같은 맥락 질문이면 → 손절
- 논쟁하지 말고 인정 + 마무리
- 시간 끌면 다른 심사위원도 지루해함

---

### 유형 4: 압박 면접 스타일 (공격형)

**상황:** "그게 말이 돼?", "이게 프로젝트라고?", "이 정도로?"

**전략: 감정 분리 + 팩트만**

**틀린 예:**
> ❌ "열심히 했는데요..." (감정적)
> ❌ "그게 뭐가 문제인데요?" (반항적)

**맞는 예:**
> ✅ (1초 멈춤) "어떤 부분이 부족해 보이셨는지 구체적으로 말씀해주시면 보완 설명드리겠습니다."

> ✅ "부족한 점이 있을 수 있습니다. 다만 **2주 개발 기간**과 **1인 프로젝트** 상황에서 **핵심 기능 구현**에 집중했습니다."

> ✅ "캡스톤 프로젝트 목적에 맞게 **실무 문제 해결**과 **학습 과정**을 보여드리려 했습니다."

**심리전:**
- 당황한 표정 ❌ → 차분한 표정 ✅
- 빨리 답변 ❌ → 1-2초 생각 후 답변 ✅
- 길게 변명 ❌ → 짧고 명확하게 ✅

### 유형 5: 전문 분야 자랑형 (본인 경험 강조)

**전략: 존중 + 차이점 설명**

**대응:**
> "**현업 경험이 풍부하신** 것 같습니다. 말씀하신 방식이 **일반적인 기업 환경**에서는 맞습니다. 다만 **공공기관 내부망**은 환경이 좀 다릅니다. [구체적 제약 설명]"

> "좋은 조언 감사합니다. 참고해서 향후 개선하겠습니다."

**포인트:**
- 심사위원 경험을 **인정**해주기
- "일반적으로는 맞지만 이 환경은 다르다"로 차별화
- 논쟁하지 말고 "참고하겠다"로 마무리

## 🏛️ 국가데이터처 환경 강조 멘트

모든 기술 선택 질문에 활용할 수 있는 **킬러 멘트**:

> "국가데이터처는 **내부망 환경**이라 인터넷 접근이 제한되고, 외부 프로그램 반입에 보안 승인이 필요합니다. 그래서 **오프라인에서 동작**하고, **의존성이 최소화**된 기술을 선택했습니다."

> "실무 환경은 **윈도우 PC + 엑셀 + 한글**입니다. 이 시스템은 그 환경에 맞춰 설계했습니다. 클라우드나 최신 기술보다 **현장에서 바로 쓸 수 있는 솔루션**이 목표였습니다."

> "기술적으로 더 좋은 방법이 있을 수 있지만, **제약 조건 안에서 최선의 선택**을 했습니다."

### 배포/출력 관련 킬러 멘트

> "Docker, 클라우드, CI/CD 다 좋지만, **내부망에서 설치 자체가 불가능**합니다. 배포는 **폴더 복사**가 전부입니다."

> "HWP 직접 생성은 **독점 포맷이라 라이브러리가 없습니다**. HTML로 생성해서 **복사-붙여넣기**로 한글에 옮기는 게 가장 현실적입니다."

> "최종 산출물이 **한글 문서**입니다. PDF는 편집이 안 되고, HTML은 복붙으로 바로 한글에 옮길 수 있어서 HTML을 선택했습니다."
