# 예상 Q&A 리스트

## 기술 관련 질문

### Q1: 스키마 기반 시스템의 장점은 무엇인가요?
**핵심 답변:**
- 새로운 보도자료 추가 시 코드 수정 없이 스키마만 작성
- 유지보수성 향상
- 데이터 매핑 명확화

**자연스러운 표현:**
"스키마 기반 시스템의 가장 큰 장점은 확장성입니다. 새로운 보도자료를 추가할 때 코드를 수정할 필요 없이 JSON 스키마 파일만 작성하면 되거든요. 이렇게 하면 유지보수가 훨씬 쉬워지고, 데이터 매핑도 명확해집니다."

---

### Q2: 가중치 기반 순위 시스템은 어떻게 작동하나요?
**핵심 답변:**
- 17개 시도의 경제지표 분석
- 증감률에 가중치 적용
- 실시간 순위 계산 및 하이라이트

**자연스러운 표현:**
"각 시도의 경제지표 증감률을 분석해서, 중요도에 따라 가중치를 부여합니다. 예를 들어 광공업생산이 서비스업생산보다 더 중요하다면 가중치를 높게 설정하는 거죠. 이렇게 해서 실시간으로 순위를 계산하고 상위/하위 지역을 자동으로 하이라이트합니다."

---

### Q3: 결측치 처리 시스템의 동작 방식을 설명해주세요.
**핵심 답변:**
- 노란색 형광펜으로 시각화
- contenteditable 필드로 직접 수정
- 모달창으로 한눈에 확인

**자연스러운 표현:**
"데이터가 누락되거나 비공개인 경우, 노란색 형광펜으로 표시합니다. 사용자는 그 부분을 클릭해서 직접 수정할 수 있어요. 그리고 모달창에서 모든 결측치를 한눈에 확인하고 일괄 처리할 수도 있습니다."

---

### Q4: 50개 보도자료를 어떻게 관리하나요?
**핵심 답변:**
- 카테고리별 분류 (요약/부문별/시도별/통계표)
- 각 보도자료마다 독립적인 스키마
- 템플릿 시스템으로 일관성 유지

**자연스러운 표현:**
"50개 보도자료를 4개 카테고리로 분류해서 관리합니다. 각 보도자료는 독립적인 JSON 스키마를 가지고 있어서, 하나를 수정해도 다른 것에 영향을 주지 않아요. 그리고 템플릿 시스템으로 일관된 포맷을 유지합니다."

---

### Q4-1: 데이터 검증은 어떻게 진행되나요? ⭐

**핵심 답변:**
- 4단계 검증 파이프라인 (파일→구조→데이터→출력)
- 자동 검증 + 수동 미리보기 검토
- 결측치 하이라이트 및 수정 기능

**자연스러운 표현:**
"데이터 검증은 4단계로 진행됩니다. 먼저 파일 업로드 시 형식과 구조를 자동으로 검증하고요, 그다음 필수 시트 존재 여부와 데이터 결측치를 확인합니다. 마지막으로 미리보기 시스템을 통해 담당자가 직접 수치를 확인하고, 검토완료 체크를 해야 최종 출력이 가능해요."

**상세 설명 (필요 시):**

```
[4단계 검증 파이프라인]

1단계: 파일 검증 (자동)
├── 파일 형식 (.xlsx, .xls)
├── 파일 유형 감지 (분석표 vs 기초자료)
└── 연도/분기 자동 추출

2단계: 구조 검증 (자동)
├── 필수 시트 9개 존재 확인
├── GRDP 시트 확인
└── 분석시트 ↔ 집계시트 매핑 검증

3단계: 데이터 검증 (자동)
├── 결측치 확인 (None, '', NaN)
├── 데이터 타입 검증
└── 범위 검증

4단계: 출력 검증 (자동 + 수동)
├── HTML 렌더링 성공 여부
├── 미리보기 담당자 확인 ← 수동
└── 검토완료 체크박스 ← 수동
```

### Q4-2: 잘못된 데이터가 입력되면 어떻게 처리되나요?

**핵심 답변:**
- 오류 유형별 차별화된 처리
- 안전한 변환 함수 (safe_float)
- 결측치는 '-'로 표시

**자연스러운 표현:**
"잘못된 데이터가 들어오면 오류 유형에 따라 다르게 처리합니다. 파일 형식이 틀리면 업로드 자체가 안 되고요, 데이터 결측은 '-'로 표시해서 담당자가 미리보기에서 확인할 수 있어요. 숫자 변환 오류는 safe_float 같은 안전한 변환 함수로 기본값을 사용하도록 했습니다."

---

### Q4-3: 왜 검증된 분석표를 사용하나요? API 직접 사용은?

**핵심 답변:**
- 분석표는 담당자가 검증한 공식 수치
- 기존 업무 프로세스와의 통합
- 신뢰성과 책임소재 명확화

**자연스러운 표현:**
"분석표를 사용하는 가장 큰 이유는 신뢰성입니다. 분석표는 담당자분들이 직접 검증한 공식 수치거든요. KOSIS API를 직접 사용하면 집계 시점이나 수정 반영 등으로 공식 발표 수치와 미세한 차이가 발생할 수 있어요. 그리고 분석표를 사용하면 기존 업무 프로세스에 자연스럽게 통합되고, 데이터 검증 책임도 명확합니다."

## 프로젝트 관련 질문

### Q5: 2주라는 짧은 기간에 어떻게 완성했나요?
**핵심 답변:**
- 단계별 개발 (Phase 1-7)
- 스키마 기반 아키텍처로 확장성 확보
- 모듈화로 병렬 개발 가능

**자연스러운 표현:**
"7단계로 나눠서 체계적으로 개발했습니다. 특히 스키마 기반 아키텍처를 도입해서, 초반에 구조를 잘 잡아두니 나중에 보도자료를 추가하는 게 훨씬 빨랐어요. 그리고 모듈화된 구조 덕분에 여러 기능을 동시에 개발할 수 있었습니다."

---

### Q6: 가장 어려웠던 부분은 무엇인가요?
**핵심 답변:**
- 의미 기반 마커 시스템 도입
- 다양한 엑셀 시트 형식 처리
- 결측치 처리 로직

**자연스러운 표현:**
"가장 어려웠던 건 의미 기반 마커 시스템이었어요. 처음엔 위치 기반으로 했는데, 엑셀 시트 형식이 조금만 바뀌어도 오류가 발생했거든요. 그래서 데이터의 의미를 파악해서 매핑하는 방식으로 전환했고, 이게 훨씬 안정적이었습니다."

---

### Q7: 실제 사용자 테스트는 했나요?
**핵심 답변:**
- 정답 데이터와 수치 일치 검증
- 다양한 시트 형식 테스트
- 국가데이터처 요구사항 반영

**자연스러운 표현:**
"네, 정답 데이터와 완전히 일치하는지 검증했습니다. 그리고 다양한 엑셀 시트 형식을 테스트해서 안정성을 확보했어요. 국가데이터처의 실제 요구사항도 반영했습니다."

## 향후 발전 관련 질문

### Q8: KOSIS API 연동은 어떻게 계획하고 있나요?
**핵심 답변:**
- 실시간 통계 데이터 자동 수집
- 기초자료 수집표 자동 생성
- 우선순위 높음

**자연스러운 표현:**
"KOSIS API를 연동하면 실시간으로 통계 데이터를 가져와서, 기초자료 수집표를 자동으로 생성할 수 있어요. 이게 완성되면 사용자가 엑셀 파일을 준비할 필요 없이, 시스템이 알아서 데이터를 수집하고 보도자료를 생성할 수 있습니다. 향후 개선 과제 중 우선순위가 가장 높은 부분입니다."

---

### Q9: 다른 기관에도 적용 가능한가요?
**핵심 답변:**
- 스키마 기반으로 확장 가능
- 템플릿 커스터마이징
- 범용적 아키텍처

**자연스러운 표현:**
"네, 가능합니다. 스키마 기반 아키텍처 덕분에 다른 기관의 보도자료 형식에도 적용할 수 있어요. 각 기관의 요구사항에 맞게 스키마와 템플릿만 수정하면 되거든요. 범용적으로 설계했습니다."

---

**핵심 답변:**

- 모듈화로 효율적 처리
- 2주 개발 기간 고려

**자연스러운 표현:**
"캐싱을 도입해서 반복적인 데이터 처리를 최소화했습니다. 그리고 모듈화된 구조로 필요한 부분만 처리하도록 했어요. 2주라는 짧은 기간이었지만, 핵심 기능은 모두 최적화했습니다."
